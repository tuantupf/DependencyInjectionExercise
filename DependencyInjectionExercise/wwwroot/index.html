Bug List

The following bugs have been reported by QA. Your task is to investigate the code, find the root cause, and fix each one.

BUG-001
Wrong discount applied - customers receive discounts they shouldn't get

The system has a "loyalty bonus" that gives an extra 5% discount on every 3rd order a customer places. However, the order counter seems to be shared across all customers. A brand new customer can receive the loyalty bonus on their very first order if other customers placed orders before them.

Steps to reproduce:
  1. Go to Place Order
  2. Place an order as "Alice" - select any book, quantity 1, place the order. Note the discount is 0%
  3. Place a second order as "Alice" - same setup, quantity 1. Discount is still 0%
  4. Now place an order as a completely new customer "Bob" - select any book, quantity 1, place the order
  5. Go to the Orders page and check Bob's order
  6. Expected: Bob's first order should have 0% discount (he is a new customer, this is his 1st order)
  7. Actual: Bob gets a 5% loyalty bonus! The order counter from Alice's orders carried over to Bob
  8. You can also verify by calling GET /api/orders/debug/discount-state in your browser - the order counter keeps incrementing across all customers
BUG-002
Order tracking note is always empty

When an order is placed, the system is supposed to generate a tracking note (e.g. "Order placed for 5x 'Dune'") and save it with the order. However, the tracking note is always null when you view the order afterwards.

Steps to reproduce:
  1. Go to Place Order and place any valid order
  2. After the order is placed, check the result - the Tracking Note field shows NULL
  3. Go to the Orders page and look at the Tracking Note column
  4. Expected: Each order should have a tracking note like "[abc123] Order placed for 3x 'Dune'"
  5. Actual: The Tracking Note column is always empty/null for every order
BUG-003
"Push" notification does not work, and adding new notification channels is too difficult

Users can select "Push" as a notification method when placing an order, but no push notification is ever sent. The order is created successfully but the notification is silently skipped. Email and SMS notifications work correctly.

Additionally, the product team plans to add more notification channels in the future (e.g. webhook, Slack). The notification system should be designed so that adding a new channel only requires creating one new class and adding one line of service registration, without modifying any existing code.

Steps to reproduce:
  1. Go to Place Order
  2. Fill in all fields and select "Push" as the notification method
  3. Place the order - the order is created successfully
  4. Call GET /api/notifications in your browser
  5. Expected: A push notification entry should appear in the notification log
  6. Actual: No notification is logged. The push channel is silently ignored
Acceptance criteria:
  1. Push notifications must work and appear in the notification log
  2. To verify extensibility: try adding a "webhook" channel after your fix - it should require only one new class and one line of registration, with zero changes to existing files
BUG-004
No separation between API layer and business logic

All business rules (validation, price calculation, discount logic, stock management, notification dispatch) are written directly inside the API controllers. There is no service layer or data access layer. This makes the code hard to test, hard to maintain, and violates clean architecture principles.

Steps to reproduce:
  1. Open BooksController.cs and OrdersController.cs
  2. Observe that the controllers directly access the database context
  3. Observe that business rules (discount calculation, stock validation, price computation) are all in the controllers
  4. Expected: Controllers should only handle HTTP request/response. Business logic should be in a service layer. Database access should be in a repository layer.
  5. Actual: Everything is mixed together in the controllers
